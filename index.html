How Next.js works

Next.js is a React framework — it gives you React plus routing, a backend, image optimization, and more, all in one project. You don't need a separate server.

my-app/ ├── src/app/ ← all your pages live here │ ├── page.tsx → yoursite.com/ │ ├── layout.tsx ← wraps every page (navbar, fonts) │ ├── about/ │ │ └── page.tsx → yoursite.com/about │ ├── blog/ │ │ ├── page.tsx → yoursite.com/blog │ │ └── [slug]/ │ │ └── page.tsx → yoursite.com/blog/any-post │ └── api/ │ └── posts/ │ └── route.ts → yoursite.com/api/posts └── public/ ← images, fonts, static files
Key rule Only files named exactly page.tsx become visible pages. Any other filename is just a helper file.
Chapter 01

JavaScript Fundamentals

Before diving into React/Next.js, you need these JS patterns — you'll use them every single day. They show up constantly in component code.

map() & filter() JS

map() transforms every item in an array and returns a new array. filter() keeps only the items that pass a test.

arrays.jsJavaScript
const nums = [1, 2, 3, 4, 5]

// map — transform each item
const doubled = nums.map(n => n * 2)
// [2, 4, 6, 8, 10]

// filter — keep items that pass the test
const evens = nums.filter(n => n % 2 === 0)
// [2, 4]

// chain them together
const result = nums
  .filter(n => n > 2)           // keep [3, 4, 5]
  .map(n => n * 10)            // double → [30, 40, 50]
page.tsxReact / Next.js
// In React, map() renders lists of JSX elements
const posts = [
  { id: 1, title: 'Hello World' },
  { id: 2, title: 'Learning Next.js' },
]

export default function Page() {
  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}

Destructuring & spread (...) JS

Destructuring unpacks values from objects/arrays. The spread operator ... copies everything — used constantly when updating React state.

destructure.jsJavaScript
// Object destructuring
const user = { name: 'Alex', age: 14, city: 'Seoul' }
const { name, age } = user   // name='Alex', age=14

// Array destructuring (useState uses this!)
const [count, setCount] = useState(0)

// Spread — copy an object and add/change fields
const updated = { ...user, age: 15 }
// { name: 'Alex', age: 15, city: 'Seoul' }

// Spread array
const newList = [...oldList, 'new item']

// In React — updating one field of state
setUser(prev => ({ ...prev, age: 15 }))
React pattern In components, destructure props directly: function Card({ title, desc }) {'{'}...{'}'} — cleaner than props.title, props.desc.

async / await & fetch() JS

Network requests take time. async/await lets you wait for them without freezing the page. fetch() is how you call APIs.

fetch-example.tsTypeScript
// async = "this function will do something that takes time"
// await = "wait here until this is done, then continue"
async function loadPosts() {
  const res  = await fetch('/api/posts')    // wait for response
  const data = await res.json()              // wait for parsing
  return data
}

// Always wrap in try/catch to handle errors
async function safeLoad() {
  try {
    const data = await loadPosts()
    setPosts(data)
  } catch (err) {
    setError('Failed to load')
  }
}

// Next.js Server Component — fetch directly in component
export default async function Page() {
  const res   = await fetch('https://api.example.com/posts')
  const posts = await res.json()
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>))}</ul>
}

Optional chaining (?.) JS

When data comes from an API, it might be null or undefined. ?. safely accesses nested properties without crashing.

optional.tsTypeScript
const user = null

// ❌ This crashes: Cannot read property 'name' of null
user.name

// ✓ Optional chaining — returns undefined instead of crashing
user?.name              // undefined (safe)
user?.address?.city    // undefined (safe, chained)

// With arrays from APIs
const { data } = await supabase.from('posts').select('*')
data?.map(post => <li>{post.title}</li>)  // safe even if data is null
Chapter 02

Pages & Routing

In Next.js, the folder structure is the routing system. Create a folder, add a page.tsx, and that's your URL — no config needed.

App Router basics Next

Every page.tsx exports a default React component. The folder name becomes the URL segment.

app/about/page.tsxTypeScript
// This file = yoursite.com/about
export default function AboutPage() {
  return (
    <main>
      <h1>About Me</h1>
      <p>Welcome to my site!</p>
    </main>
  )
}

Special filenames Next.js recognizes:

page.tsx ← the page itself layout.tsx ← wraps all pages in this folder loading.tsx ← shown while data loads error.tsx ← shown if something crashes not-found.tsx ← shown for 404 errors route.ts ← API endpoint (no UI)

Link & navigation Next

Use <Link> instead of <a>. It prefetches pages and navigates instantly without a full reload.

❌ Plain <a> tag

  • Full page reload every time
  • Slow, loses React state
  • No prefetching

✓ Next.js <Link>

  • Instant navigation
  • Prefetches on hover
  • Keeps React state
Navbar.tsxTypeScript
import Link from 'next/link'

export default function Navbar() {
  return (
    <nav>
      <Link href="/">Home</Link>
      <Link href="/about">About</Link>
      <Link href="/blog">Blog</Link>
    </nav>
  )
}

// Programmatic navigation — after form submit, etc.
'use client'
import { useRouter } from 'next/navigation'

const router = useRouter()
router.push('/dashboard')   // navigate programmatically

Dynamic routes [slug] Next

Wrap a folder name in [brackets] to match any URL segment. One file handles /blog/hello, /blog/my-post, etc.

app/blog/[slug]/page.tsxTypeScript
// Next.js passes the URL segment via the "params" prop
export default function BlogPost({
  params,
}: {
  params: { slug: string }
}) {
  // URL: /blog/hello-world  →  params.slug = "hello-world"
  return <h1>Post: {params.slug}</h1>
}

// In real app: use params.slug to fetch from DB
const { data } = await supabase
  .from('posts')
  .select('*')
  .eq('slug', params.slug)
  .single()

layout.tsx Next

A layout.tsx wraps every page in its folder — and in nested folders too. Use it for navbars, footers, fonts.

app/layout.tsxTypeScript

  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body>
        <Navbar />
        {children}    {'/* every page renders here */'}
        <Footer />
      </body>
    </html>
  )
}
Chapter 03

Events & State

Static pages are boring. Events let you respond to user actions — clicking, typing, submitting. State lets you store values that change and re-render the UI automatically.

Server vs Client Components Next

This is the most important concept in App Router. By default everything is a Server Component. Add 'use client' to the top of a file to make it interactive.

Server Component (default)

  • Runs on the server
  • Can fetch from database
  • No useState, onClick, useEffect
  • Faster — no JS sent to browser

Client Component ('use client')

  • Runs in the browser
  • Can use useState, events, refs
  • Cannot directly query a database
  • Needed for any interactivity
Simple rule If your component needs onClick, onChange, useState, or useEffect — add 'use client' as the very first line.

useState JS

useState stores a value that can change. When you update it, React automatically re-renders the component.

Counter.tsxTypeScript
'use client'
import { useState } from 'react'

export default function Counter() {
  // [currentValue, updateFunction] = useState(initialValue)
  const [count, setCount] = useState(0)

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>−</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  )
}

onClick & events JS

Event handlers are functions you pass to elements. They run when the user interacts.

events.tsxTypeScript
// Inline handler
<button onClick={() => console.log('clicked!')}>Click me</button>

// Named handler (cleaner for complex logic)
function handleClick() {
  setLoading(true)
  // do something...
  setLoading(false)
}
<button onClick={handleClick}>Submit</button>

// Common events
onClick    // click / tap
onChange   // input value changed
onSubmit   // form submitted
onKeyDown  // keyboard key pressed

Forms & onChange JS

In React, inputs are controlled — their value always comes from state. onChange updates state as the user types.

ContactForm.tsxTypeScript
'use client'
import { useState } from 'react'

export default function ContactForm() {
  const [form, setForm] = useState({
    name:  '',
    email: '',
  })

  // Update one field, keep the rest (spread pattern)
  function update(field: string, value: string) {
    setForm(prev => ({ ...prev, [field]: value }))
  }

  return (
    <form>
      <input
        value={form.name}
        onChange={e => update('name', e.target.value)}
        placeholder="Your name"
      />
      <input
        value={form.email}
        onChange={e => update('email', e.target.value)}
        placeholder="Email address"
      />
    </form>
  )
}

useEffect JS

useEffect runs code after the component renders — good for fetching data on load, setting up timers, or syncing with external things.

PostList.tsxTypeScript
'use client'
import { useState, useEffect } from 'react'

export default function PostList() {
  const [posts,   setPosts]   = useState([])
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    // Runs once when the component mounts
    async function load() {
      const res  = await fetch('/api/posts')
      const data = await res.json()
      setPosts(data)
      setLoading(false)
    }
    load()
  }, [])  // ← empty array = run once on mount

  if (loading) return <p>Loading...</p>
  return <ul>{posts.map(p => <li key={p.id}>{p.title}</li>))}</ul>
}
Tip Prefer Server Components + async/await fetch for loading data in Next.js — it's simpler and faster. Use useEffect only in Client Components when you must.

Rendering lists JS

Use .map() to turn an array into JSX. Always add a key prop — React uses it to track which items changed.

TodoList.tsxTypeScript
'use client'
import { useState } from 'react'

export default function TodoList() {
  const [todos, setTodos] = useState(['Buy milk', 'Do homework'])

  // Add item
  function add(text: string) {
    setTodos(prev => [...prev, text])
  }

  // Remove item by index
  function remove(i: number) {
    setTodos(prev => prev.filter((_, idx) => idx !== i))
  }

  return (
    <ul>
      {todos.map((todo, i) => (
        <li key={i}>
          {todo}
          <button onClick={() => remove(i)}>✕</button>
        </li>
      ))}
    </ul>
  )
}

Chapter 04

Backend

Next.js lets you write backend code inside your project — no separate server. Route Handlers are API endpoints, Server Actions are functions that run on the server.

How the web works BE

1
Request — your browser sends an HTTP request to a server (a computer somewhere in the world)
2
Server logic — the server runs code, maybe queries a database, builds a response
3
Response — the server sends back HTML, JSON, or a file
4
Render — your browser shows the HTML, or React processes the JSON

HTTP methods tell the server what you want to do:

GET Read data — "give me all posts"
POST Send new data — "create a post"
DELETE Remove data — "delete post #5"
PATCH Update data — "edit the title of post #5"

Route Handlers BE

Create route.ts inside app/api/. Export functions named GET, POST, etc. These become real API endpoints.

app/api/posts/route.tsTypeScript
import { NextResponse } from 'next/server'

// GET /api/posts
export async function GET() {
  const posts = [
    { id: 1, title: 'Hello World' },
  ]
  return NextResponse.json(posts)
}

// POST /api/posts  — creates a new post
export async function POST(req: Request) {
  const body = await req.json()
  // save body to database...
  return NextResponse.json({ success: true })
}

// DELETE /api/posts/[id]  →  app/api/posts/[id]/route.ts
export async function DELETE(
  req: Request,
  { params }: { params: { id: string } }
) {
  // delete row where id = params.id
  return NextResponse.json({ deleted: true })
}

Server Actions BE

Mark a function with 'use server' and call it directly from a component — no API route needed. Cleaner, less boilerplate.

app/actions.tsTypeScript
'use server'  // whole file runs on server

import supabase from '@/lib/supabase'

export async function createPost(title: string) {
  const { data, error } = await supabase
    .from('posts')
    .insert({ title })
    .select()
    .single()
  if (error) throw new Error(error.message)
  return data
}
app/new-post/page.tsxTypeScript
'use client'
import { createPost } from '../actions'
import { useState } from 'react'

export default function NewPost() {
  const [title,   setTitle]   = useState('')
  const [loading, setLoading] = useState(false)

  async function submit() {
    setLoading(true)
    const post = await createPost(title)  // calls server!
    setLoading(false)
    console.log('created:', post)
  }

  return (
    <div>
      <input value={title} onChange={e => setTitle(e.target.value)} />
      <button onClick={submit} disabled={loading}>
        {loading ? 'Saving...' : 'Publish'}
      </button>
    </div>
  )
}

Environment variables BE

Never put secrets (API keys, passwords) in your code. Store them in .env.local — this file stays on your machine and never gets pushed to GitHub.

.env.localenv
# NEXT_PUBLIC_ prefix = accessible in browser too
NEXT_PUBLIC_SUPABASE_URL=https://xxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGci...

# No prefix = server only (NEVER sent to browser)
SUPABASE_SERVICE_ROLE_KEY=eyJhbGci...
DATABASE_PASSWORD=super-secret-123
Critical Add .env.local to your .gitignore file. If you accidentally push it to GitHub, rotate (regenerate) all your keys immediately.

CRUD operations BE

CRUD = Create, Read, Update, Delete — the four things you do with any data.

db-operations.tsTypeScript
import supabase from '@/lib/supabase'

// READ — get all posts, newest first
const { data } = await supabase
  .from('posts')
  .select('*')
  .order('created_at', { ascending: false })

// READ — one post by id
const { data: post } = await supabase
  .from('posts')
  .select('*')
  .eq('id', 5)
  .single()

// CREATE — insert a new row
await supabase
  .from('posts')
  .insert({ title: 'New post', content: 'Hello!' })

// UPDATE — change fields on a row
await supabase
  .from('posts')
  .update({ title: 'Updated title' })
  .eq('id', 5)

// DELETE — remove a row
await supabase
  .from('posts')
  .delete()
  .eq('id', 5)

Vibe Coding with Cursor

Cursor is a code editor with AI built in. The real skill isn't knowing every API — it's knowing how to describe what you want clearly enough that AI can help you build it.

What is Cursor? AI

⌘
Chat — Cmd+L (Mac) / Ctrl+L (Win)
Open a chat sidebar. Ask questions, request code, paste errors. Cursor reads your whole project.
✏️
Inline edit — Cmd+K (Mac) / Ctrl+K (Win)
Select code, press Cmd+K, describe the change. Cursor rewrites just that selection.
🐛
Debug — paste your error
Copy the exact error from your terminal or browser. Paste it in chat. Cursor finds the bug.
1
Download at cursor.com — free tier available
2
Open your Next.js project folder in Cursor
3
Press Cmd+L to open chat — start asking

Prompt formula AI

Every good prompt has the same 5 parts. Skip one and the output quality drops noticeably.

1
Stack: "I'm using Next.js 14 App Router with TypeScript"
2
File: "In app/posts/page.tsx"
3
What exists: "I have a list of posts rendered with .map()"
4
Goal: "Add a delete button that calls DELETE /api/posts/[id]"
5
Edge cases: "Show loading spinner while deleting, remove from list on success"

Good vs bad prompts AI

❌ Bad

add a button that gets data

✓ Good

Next.js 14 App Router, TypeScript. In app/page.tsx (Client Component), add a "Load Posts" button. On click: fetch GET /api/posts, store result in useState, render as a <ul> list. Show "Loading..." while fetching.

❌ Bad

my app is broken fix it

✓ Good

Next.js 14, TypeScript. Error: "Cannot read properties of undefined (reading 'map')" Line 14 in app/page.tsx. Here's the component: [paste code] API /api/posts returns: [paste JSON]

❌ Bad

make a form

✓ Good

Next.js 14, TypeScript, CSS Modules. Create components/CreatePostForm.tsx. Fields: title (required), content (textarea). On submit: call createPost() Server Action. States: idle → loading ("Saving...") → success ("Saved!").

Workflow AI

1
Plan first. Write what you want in plain English. Break it into small pieces — one component at a time.
2
Set context once. Tell Cursor your stack at the start of a conversation. It applies to everything after.
3
Build one thing. One component per prompt. Review + test before the next one.
4
Iterate, don't restart. "It works but the date format is wrong" — keep the same chat thread.
5
Ask to explain. "What does line 12 do?" — never ship code you don't understand.
Golden rule AI is a fast coder but a bad decision-maker. You decide what to build. AI writes the code. Always read the output before using it.
